# v0.3.0 — Qwen / vLLM Model Gateway and Deployment Profile

Release date: 2026-07-10 UTC

## Scope

Module 1 replaces the explicit production-model stub with a provider-neutral,
fail-closed adapter for a privately served Qwen model behind vLLM's
OpenAI-compatible API. It also adds a deployable Compose profile and the exact
offline/live verification boundary needed to operate that service responsibly.

This is a source release of the gateway and deployment assets. Live model load,
GPU inference and hardware performance evidence are `NOT_RUN`; this release does
not claim that a Qwen checkpoint was running on the development host.

## Implemented

- contract-v2 assistant tool-call replay for multi-turn function workflows;
- non-streaming text, multiple tool calls and strict structured JSON output;
- local tool-name, argument-schema and structured-output validation before any
  value can reach an MCP executor;
- ordered SSE text/tool events with exactly one terminal `completed` or `error`;
- explicit unknown-token-usage sentinel instead of invented counts;
- bounded exponential retry, `Retry-After`, total request/stream deadlines and
  no retry after visible streamed output;
- pooled `httpx.Client` lifecycle with TLS verification, ambient proxy disabled,
  redirects disabled and provider objects contained inside the gateway;
- strict JSON duplicate-key, non-finite-number, nesting and node-count limits;
- exact configured served-model ID, one index-zero choice and unique tool-call ID
  enforcement;
- Qwen3/vLLM Compose profile, safe Bash-array entrypoint, health check, plain/tool/
  structured/SSE smoke checks, benchmark tool and GPU sizing worksheet;
- deployment security, admission, observability, canary and rollback runbook;
- environment-gated live integration suite requiring an explicit private endpoint.

## Operational deployment boundary

Live status: `NOT_RUN`.

The release environment had no approved idle GPU allocation or installed Docker/
vLLM runtime for a non-interfering model launch. Therefore latency, TTFT, TPOT,
throughput, GPU memory and live tool/structured validity rates are intentionally
absent. Complete `docs/qwen-deployment-runbook.md` and
`deploy/qwen-vllm/gpu-sizing-worksheet.md` on the target host before calling the
private deployment operational.

Known limitations:

- production gateway startup requires the `httpx` implementation dependency;
- enterprise HTTPS with a private CA requires an explicit
  `VllmGatewayConfig.ca_bundle_path` or a trusted internal TLS gateway; public
  certificates use the HTTP client's default CA bundle, and verification cannot
  be disabled;
- only one configured served-model ID and one completion choice are accepted;
- live Qwen behavior still depends on the exact checkpoint, chat template, tool
  parser, reasoning parser and vLLM image combination;
- this module does not perform MCP execution, RAG, LangGraph orchestration or
  domain-algorithm computation.

These limits preserve a clear interview explanation: the repository proves the
gateway protocol and deployment procedure, while hardware-specific claims require
separate measured evidence.
